Cos'è Docker?

Docker è una piattaforma open source che automatizza il deployment di applicazioni all'interno di container software. I container consentono di incapsulare un'applicazione con tutte le sue dipendenze (librerie, file di configurazione, variabili d'ambiente) in un'unica unità eseguibile standardizzata.

A differenza delle macchine virtuali tradizionali, i container condividono il kernel del sistema operativo host, risultando molto più leggeri e veloci da avviare. Questo approccio risolve il problema del "funziona sulla mia macchina" garantendo coerenza tra ambienti di sviluppo, testing e produzione.

Perché usare Docker?

  • Isolamento - Ogni applicazione gira in un ambiente isolato
  • Portabilità - Esegui lo stesso container ovunque
  • Scalabilità - Facile scalare orizzontalmente le applicazioni
  • Efficienza - Minore consumo di risorse rispetto alle VM
  • Versioning - Tracking delle versioni delle immagini
  • Ecosistema - Vasto marketplace di immagini pre-costruite

Storia ed Evoluzione di Docker

2010 - I Precursori

Le tecnologie di containerizzazione esistevano già prima di Docker. LXC (Linux Containers) forniva le basi per l'isolamento a livello di sistema operativo, ma richiedeva una configurazione complessa e non offriva un formato standard per le applicazioni.

2013 - Nascita di Docker

Solomon Hykes presenta Docker alla conferenza PyCon come progetto interno di dotCloud, una piattaforma PaaS. L'idea era di creare uno strumento che semplificasse la distribuzione di applicazioni sfruttando la containerizzazione, ma con un approccio più user-friendly.

2014 - Docker diventa Open Source

Docker viene rilasciato come progetto open source e rapidamente guadagna popolarità nella comunità developer. Viene introdotto Docker Hub, il registry pubblico per condividere immagini Docker.

2015 - Standardizzazione e Ecosistema

Nascono standard come OCI (Open Container Initiative) e CNCF (Cloud Native Computing Foundation). Viene introdotto Docker Compose per orchestrare applicazioni multi-container.

2017-oggi - Enterprise e Kubernetes

Docker si espande nel mondo enterprise con Docker Enterprise Edition. L'ecosistema si evolve con l'ascesa di Kubernetes come standard de facto per l'orchestrazione di container, e Docker integra sempre più funzionalità per supportare questo ecosistema.

Architettura di Docker

Docker Client

Interfaccia a riga di comando (CLI) che permette agli utenti di interagire con Docker. Il client invia comandi al Docker Daemon tramite l'API REST.

Docker Daemon

Il motore di Docker che gira in background e gestisce container, immagini, reti e volumi. Ascolta le richieste del client e le elabora.

Docker Registry

Repository per archiviare e distribuire immagini Docker. Docker Hub è il registry pubblico ufficiale, ma è possibile utilizzare registry privati.

Componenti Kernel Linux

  • Namespaces - Forniscono isolamento per processi, reti, filesystem, etc.
  • Control Groups (cgroups) - Limitano e monitorano l'uso di risorse (CPU, memoria, I/O)
  • Union File Systems - Permettono la creazione di immagini a strati leggere
  • Container Runtime - Implementazioni come runc che eseguono effettivamente i container

Immagini e Container

Un'immagine Docker è un template di sola lettura che contiene le istruzioni per creare un container. È composta da una serie di layer, ognuno dei quali rappresenta una modifica al filesystem.

Un container è un'istanza eseguibile di un'immagine. Aggiunge un layer scrivibile sopra l'immagine di base, permettendo l'esecuzione dell'applicazione.

Container Docker vs Macchine Virtuali

Differenze Architetturali Fondamentali

Le macchine virtuali virtualizzano l'hardware, richiedendo un sistema operativo completo per ogni istanza (guest OS). Questo comporta overhead significativo in termini di risorse.

I container Docker virtualizzano a livello di sistema operativo, condividendo il kernel dell'host. Ogni container esegue come processo isolato, risultando molto più leggero ed efficiente.

Caratteristica Docker Container Macchine Virtuali
Tempo di avvio Secondi o millisecondi Minuti
Dimensioni MB (solo app e dipendenze) GB (SO completo incluso)
Densita Alta - centinaia per host Bassa - 10-20 per host
Isolamento Processo a livello di OS Completo a livello di hardware
Portabilità Alta - stesso container ovunque Limitata - dipende dall'hypervisor
Sicurezza Namespace e cgroups Hypervisor completo

Dockerfile: Creare Immagini Personalizzate

Cos'è un Dockerfile?

Un Dockerfile è un file di testo che contiene tutte le istruzioni necessarie per costruire un'immagine Docker. Definisce:

  • L'immagine di base da cui partire
  • I comandi da eseguire durante il build
  • I file da copiare nell'immagine
  • Le variabili d'ambiente da impostare
  • Il comando di default da eseguire

Istruzioni Principali

  • FROM - Specifica l'immagine base
  • RUN - Esegue comandi durante il build
  • COPY/ADD - Copia file nell'immagine
  • WORKDIR - Imposta la directory di lavoro
  • ENV - Imposta variabili d'ambiente
  • EXPOSE - Documenta le porte da esporre
  • CMD/ENTRYPOINT - Specifica il comando di avvio

Esempio di Dockerfile

# Utilizza un'immagine base di Node.js
FROM node:16-alpine

# Imposta la directory di lavoro
WORKDIR /app

# Copia il package.json e package-lock.json
COPY package*.json ./

# Installa le dipendenze
RUN npm ci --only=production

# Copia il codice sorgente
COPY . .

# Espone la porta 3000
EXPOSE 3000

# Definisce il comando di avvio
CMD ["node", "app.js"]

Best Practices per Dockerfile

  • Usa immagini base ufficiali e leggere
  • Minimizza il numero di layer (combina comandi RUN)
  • Usa .dockerignore per escludere file non necessari
  • Ordina le istruzioni dal meno al più frequente cambiamento
  • Utilizza utenti non-root quando possibile

Docker Compose: Orchestrazione Multi-Container

Cos'è Docker Compose?

Docker Compose è uno strumento che permette di definire e gestire applicazioni multi-container attraverso un file YAML. Con un singolo comando, è possibile avviare tutti i servizi di un'applicazione.

Vantaggi di Docker Compose

  • Definizione centralizzata di tutti i servizi
  • Facile configurazione di reti e volumi condivisi
  • Gestione semplificata delle dipendenze tra servizi
  • Riproducibilità degli ambienti di sviluppo
  • Testing di applicazioni multi-container

Comandi Principali

docker-compose up -d
Avvia tutti i servizi in background
docker-compose down
Ferma e rimuove tutti i servizi
docker-compose ps
Mostra lo stato dei servizi
docker-compose logs
Visualizza i log di tutti i servizi

Esempio di docker-compose.yml

version: '3.8'

services:
web:
build: .
ports:
- "3000:3000"
environment:
- NODE_ENV=production
depends_on:
- db

db:
image: postgres:13
environment:
POSTGRES_DB: myapp
POSTGRES_USER: user
POSTGRES_PASSWORD: password
volumes:
- db_data:/var/lib/postgresql/data
ports:
- "5432:5432"

volumes:
db_data:

Casi d'Uso Comuni

  • Applicazioni web con database
  • Stack LAMP/LEMP (Linux, Apache/Nginx, MySQL, PHP)
  • Microservizi che comunicano tra loro
  • Ambienti di sviluppo con servizi di supporto
  • Testing di integrazione con servizi mock

Best Practices e Sicurezza

Sicurezza dei Container

Principio del Privilegio Minimo

Esegui i container con il minor numero di privilegi possibile. Evita di eseguire container come root quando non necessario.

Immagini Verificate

Utilizza solo immagini da fonti attendibili e verificate. Le immagini ufficiali su Docker Hub sono generalmente più sicure.

Segreti e Configurazioni

Non includere mai segreti (password, API keys) direttamente nelle immagini. Utilizza invece i Docker Secrets o variabili d'ambiente.

Ottimizzazione delle Immagini

Immagini Minimali

Utilizza immagini base minimali (Alpine Linux, distroless) per ridurre la superficie di attacco e le dimensioni.

Ottimizzazione dei Layer

Combina comandi RUN per ridurre il numero di layer e ordina le istruzioni dal meno al più frequente cambiamento.

Multi-Stage Build

Utilizza multi-stage build per separare l'ambiente di build da quello di runtime, mantenendo le immagini finali leggere.

Monitoraggio e Logging

Metriche delle Prestazioni

Monitora CPU, memoria, I/O e rete dei container

Centralizzazione Log

Invia i log a sistemi centralizzati (ELK, Splunk)

Health Check

Implementa health check per monitorare lo stato delle applicazioni
